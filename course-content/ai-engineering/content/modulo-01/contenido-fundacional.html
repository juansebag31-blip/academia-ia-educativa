1. Propósito del módulo

Este módulo establece el lenguaje común que utilizaremos durante todo el curso. Antes de estudiar RAG, herramientas, MCP, memoria, evaluación o sistemas multiagente, necesitamos distinguir con precisión qué es cada elemento y qué función cumple dentro de una solución de inteligencia artificial.

En el uso cotidiano, términos como modelo, asistente, agente y automatización suelen mezclarse. Esa confusión puede parecer menor cuando estamos experimentando, pero se vuelve problemática cuando debemos diseñar un producto, estimar costes, explicar una arquitectura en una entrevista o garantizar que un sistema no realice acciones indebidas.

La meta no es memorizar definiciones aisladas. La meta es adquirir una forma de pensar: observar un problema real, identificar su grado de incertidumbre, decidir qué partes pueden resolverse mediante código tradicional, cuáles necesitan un modelo, cuáles requieren recuperación de información y cuáles justifican que un agente tome decisiones dinámicas.

Idea central Un modelo fundacional es una capacidad dentro del sistema. El producto real surge cuando esa capacidad se combina con interfaz, datos, herramientas, reglas, evaluación, seguridad y supervisión humana.

Objetivos de aprendizaje

  • Diferenciar modelo, aplicación de IA, workflow, agente y sistema multiagente.
  • Reconocer los componentes que convierten una respuesta probabilística en un servicio útil.
  • Comprender por qué una arquitectura más compleja no es necesariamente una arquitectura mejor.
  • Relacionar cada concepto con procesos productivos verificables.
  • Diseñar la primera versión del JSG AI Engineering Hub.

2. Panorama general: cinco niveles de construcción

Podemos imaginar la evolución de una solución de IA como una secuencia de cinco niveles. Cada nivel incorpora capacidades y también nuevas responsabilidades.

Modelo→Aplicación de IA→Workflow→Agente→Sistema multiagente

La flecha no significa que todos los productos deban recorrer obligatoriamente los cinco niveles. Muchas soluciones valiosas terminan correctamente en el segundo o en el tercero. Un buscador documental con buenas fuentes, citas y evaluación puede ser superior a un sistema multiagente innecesariamente complejo.

Anthropic propone una distinción arquitectónica especialmente útil: los workflows siguen caminos predefinidos por código, mientras que los agentes permiten que el modelo dirija dinámicamente su propio proceso y la utilización de herramientas. La misma guía recomienda comenzar por la solución más sencilla y aumentar la complejidad únicamente cuando exista una mejora demostrable en los resultados.[1]

3. Qué es un modelo de inteligencia artificial

Un modelo fundacional es un sistema entrenado con grandes cantidades de datos para aprender patrones generales que luego pueden adaptarse a numerosas tareas. Los modelos de lenguaje reciben una secuencia de entrada y producen una continuación probable. Aunque desde la experiencia del usuario parezcan “comprender” una petición, su funcionamiento no equivale a poseer por defecto conocimiento completo del negocio, acceso a documentos privados o responsabilidad sobre las consecuencias de una acción.

Modelo
Componente computacional que transforma una entrada en una salida según patrones aprendidos durante el entrenamiento y las instrucciones presentes en el contexto actual.

3.1 Lo que un modelo sí puede aportar

Dependiendo de sus capacidades, un modelo puede redactar, resumir, clasificar, traducir, extraer entidades, interpretar instrucciones, generar código, elegir entre herramientas disponibles y producir salidas estructuradas. También puede combinar información presente en el contexto para elaborar una respuesta nueva.

3.2 Lo que un modelo no conoce automáticamente

El modelo no conoce por defecto el estado actualizado de una organización, las reglas internas, los permisos de cada usuario ni el contenido de una base documental privada. Tampoco sabe automáticamente si una respuesta es aceptable para el negocio. Para adquirir ese conocimiento operativo necesita que la aplicación le entregue contexto, fuentes, herramientas y criterios de evaluación.

3.3 Probabilidad no significa arbitrariedad

Decir que la salida es probabilística no significa que el modelo responda al azar sin estructura. Significa que existe variabilidad y que la respuesta se genera a partir de distribuciones aprendidas, no de una base de datos determinista que garantice una única salida correcta. Por eso el diseño profesional no confía únicamente en la fluidez del texto: añade fuentes, validaciones, esquemas, pruebas y controles.

Error frecuente Confundir una respuesta convincente con una respuesta verificada. La calidad lingüística es una propiedad distinta de la exactitud factual, la vigencia de la información y el cumplimiento de una política empresarial.

3.4 El modelo como “motor”, no como “vehículo completo”

Una analogía útil es pensar en el modelo como el motor de un vehículo. El motor aporta potencia, pero no define por sí solo el volante, los frenos, las normas de circulación, la ruta, el tablero de control ni la responsabilidad del conductor. Del mismo modo, el modelo aporta capacidades cognitivas aproximadas, pero la aplicación debe proporcionar control, dirección y seguridad.

4. De un modelo a una aplicación de IA

Una aplicación de inteligencia artificial es un sistema de software que utiliza uno o más modelos para resolver una necesidad concreta. La diferencia decisiva está en que la aplicación incorpora contexto operativo y transforma la capacidad general del modelo en una experiencia repetible.

Aplicación de IA
Producto de software que combina modelos con interfaz, lógica de negocio, datos, herramientas, persistencia, seguridad y mecanismos de evaluación.

4.1 El recorrido básico de una solicitud

  1. El usuario expresa una necesidad mediante una interfaz.
  2. El backend autentica al usuario y determina qué acciones puede realizar.
  3. La aplicación prepara el contexto: instrucciones, datos, historial y documentos relevantes.
  4. El modelo procesa la solicitud y puede devolver texto, datos estructurados o una petición de herramienta.
  5. La aplicación valida el resultado y decide si debe mostrarse, corregirse, registrarse o solicitar aprobación.
  6. La respuesta y los eventos de la ejecución quedan disponibles para evaluación y mejora.

4.2 Por qué el backend es esencial

El backend no es simplemente un puente técnico entre la interfaz y el modelo. Es el lugar donde se aplican las reglas: quién puede consultar un documento, qué herramienta está permitida, cuánto puede gastar una tarea, qué información debe anonimizarse y qué operaciones requieren confirmación.

4.3 La interfaz también forma parte de la inteligencia del sistema

Una interfaz bien diseñada evita peticiones ambiguas, muestra las fuentes, diferencia una propuesta de una acción ejecutada y permite que el usuario corrija el rumbo. En AI Engineering, la calidad no depende solamente del prompt; también depende de cómo la aplicación presenta opciones, límites y evidencia.

Decisión arquitectónica En este curso separaremos siempre la capa de experiencia, la capa de orquestación, la capa de conocimiento y la capa de acciones. Esa separación facilita la actualización del modelo sin reconstruir todo el producto.

5. Workflows: inteligencia dentro de un proceso predefinido

Un workflow es una secuencia organizada de pasos. Algunos pueden estar resueltos mediante código tradicional y otros mediante modelos. Lo importante es que el camino general está definido previamente.

Workflow
Sistema en el que modelos y herramientas son coordinados mediante rutas de ejecución determinadas por el software.

5.1 Cuándo conviene utilizar un workflow

Los workflows son adecuados cuando el proceso es repetitivo, las etapas son conocidas y necesitamos consistencia. Por ejemplo, recibir un documento, verificar su tipo, extraer campos, validar formatos, consultar una tabla y generar un informe.

5.2 Patrones frecuentes

Anthropic describe varios patrones productivos: encadenamiento de prompts, enrutamiento, paralelización, orquestador con trabajadores y ciclos de evaluador-optimizador.[1]

Encadenamiento

Divide una tarea en etapas. La salida de una etapa se convierte en entrada de la siguiente. Es útil cuando cada paso puede verificarse y la división reduce la dificultad.

Enrutamiento

Clasifica la petición y la dirige hacia un procedimiento especializado. Puede utilizarse para diferenciar consultas administrativas, técnicas o sensibles, o para enviar tareas simples a un modelo económico y tareas complejas a uno más capaz.

Paralelización

Ejecuta tareas independientes al mismo tiempo. También puede solicitar varias evaluaciones sobre el mismo resultado y combinar sus conclusiones.

Orquestador y trabajadores

Un componente central divide el trabajo y lo distribuye. A diferencia de una paralelización completamente fija, la cantidad y naturaleza de las subtareas puede depender de la solicitud recibida.

Evaluador y optimizador

Un modelo genera una respuesta y otro proceso la evalúa según criterios definidos. El ciclo continúa hasta alcanzar un nivel aceptable o un límite de iteraciones.

Ejemplo productivo Una clínica recibe una orden médica. El workflow identifica el estudio, extrae región anatómica y contraste, consulta la tabla vigente, verifica incompatibilidades y prepara una propuesta para revisión administrativa. La secuencia es conocida; por eso no necesitamos que un agente invente libremente el proceso.

6. Agentes: autonomía limitada para problemas abiertos

Un agente es un sistema donde el modelo puede decidir qué pasos realizar y qué herramientas utilizar para alcanzar un objetivo. Esa autonomía no debe interpretarse como libertad irrestricta. El agente opera dentro de un entorno diseñado por desarrolladores, con herramientas, permisos, límites y condiciones de detención.

Agente
Modelo equipado con instrucciones, herramientas y un ciclo de ejecución que le permite observar resultados, decidir el siguiente paso y continuar hasta completar o detener una tarea.

La documentación del OpenAI Agents SDK organiza este tipo de aplicaciones alrededor de primitivas como agentes, herramientas, handoffs, sesiones, guardrails, intervención humana y trazabilidad.[2] Esto muestra que un agente profesional no se reduce a “un prompt autónomo”; requiere infraestructura para ejecutar y observar su comportamiento.

6.1 El ciclo básico del agente

  1. Recibe un objetivo y el contexto inicial.
  2. Analiza el estado y decide una acción.
  3. Invoca una herramienta o produce una respuesta intermedia.
  4. Observa el resultado real de la herramienta.
  5. Actualiza su plan y continúa.
  6. Finaliza al completar el objetivo, alcanzar un límite o requerir intervención humana.

6.2 Ground truth: la realidad debe responder

Un agente confiable necesita información del entorno. Si modifica código, debe ejecutar pruebas. Si consulta un calendario, debe recibir la respuesta real del servicio. Si busca documentación, debe poder citar el fragmento recuperado. La retroalimentación externa permite detectar errores que el modelo no podría descubrir mediante reflexión textual aislada.

6.3 Cuándo un agente aporta valor

Conviene considerar un agente cuando el número de pasos no puede predecirse fácilmente, las herramientas dependen del contexto y el sistema necesita adaptarse a resultados intermedios. La resolución de incidencias de software es un ejemplo: el agente inspecciona archivos, formula una hipótesis, modifica código, ejecuta pruebas y corrige según los fallos.

6.4 Coste de la autonomía

Cada iteración consume tiempo y recursos. Además, un error inicial puede influir en decisiones posteriores. Por eso los agentes necesitan límites de pasos, presupuestos, herramientas claramente definidas, entornos aislados y puntos de aprobación.

Principio de seguridad Nunca debemos otorgar una capacidad irreversible solo porque el modelo “parece entender” la tarea. El nivel de autonomía debe aumentar junto con la capacidad de verificar, registrar y revertir acciones.

7. Sistemas multiagente: especialización con un coste de coordinación

Un sistema multiagente distribuye responsabilidades entre varias unidades especializadas. Puede existir un supervisor central, transferencias entre agentes o colaboración paralela. La especialización puede mejorar claridad y calidad, pero también aumenta el número de interacciones y los puntos de fallo.

Multiagente
Arquitectura en la que varios agentes, con funciones o herramientas diferentes, colaboran para resolver una tarea común.

7.1 Razones válidas para separar agentes

  • Los dominios requieren instrucciones y herramientas muy diferentes.
  • Existen permisos incompatibles que conviene aislar.
  • Varias tareas pueden ejecutarse en paralelo.
  • Se necesita una revisión independiente del resultado.
  • El contexto completo sería demasiado grande o confuso para un único agente.

7.2 Razones inválidas

No deberíamos crear cinco agentes únicamente porque sus nombres resultan atractivos. Si todos reciben casi las mismas instrucciones y utilizan las mismas herramientas, probablemente estamos añadiendo capas sin ganar especialización real.

7.3 Orquestación y handoffs

La orquestación define quién conserva el control. En un modelo supervisor, un agente central delega y reúne resultados. En un handoff, un agente transfiere la conversación o la tarea a otro especializado. OpenAI documenta ambas posibilidades dentro de su SDK y ofrece mecanismos para registrar la ejecución completa.[2]

Regla del curso El JSG AI Engineering Hub comenzará con un único orquestador. Solo separaremos agentes cuando podamos escribir una justificación concreta relacionada con especialización, permisos, paralelismo o evaluación independiente.

8. Componentes fundamentales de un sistema inteligente

8.1 Interfaz de usuario

Recoge la intención, presenta resultados, evidencia incertidumbre y permite confirmar acciones. Puede ser una aplicación web, móvil, chat, voz o integración dentro de otra herramienta.

8.2 Backend y orquestación

Gestiona autenticación, reglas, contexto, llamadas al modelo, herramientas, errores, reintentos y persistencia. Es el centro de control del producto.

8.3 Herramientas y APIs

Permiten que el sistema consulte o modifique el mundo externo. Una buena herramienta posee nombre claro, parámetros comprensibles, validación, permisos y respuestas estructuradas. Anthropic destaca que la documentación de herramientas merece tanta atención como el prompt general del agente.[1]

8.4 Datos y conocimiento

Incluyen documentos, bases de datos, reglas de negocio y fuentes externas. En el módulo 5 estudiaremos cómo recuperar fragmentos relevantes mediante RAG, mantener versiones y citar evidencia.

8.5 Contexto, estado y memoria

El contexto es la información disponible en una llamada. El estado representa la situación actual de una tarea. La memoria permite conservar información más allá de una interacción. Separar estos conceptos evita introducir datos irrelevantes o mantener información que debería expirar.

8.6 Guardrails y permisos

Son controles sobre entradas, salidas y herramientas. Pueden bloquear solicitudes, validar formatos, exigir aprobación o impedir acciones fuera del alcance autorizado.

8.7 Observabilidad y trazabilidad

Permiten reconstruir qué ocurrió: qué instrucciones recibió el sistema, qué herramienta eligió, cuánto tardó, cuánto costó y por qué terminó. El OpenAI Agents SDK incluye trazabilidad como una capacidad de primera clase.[2]

8.8 Evaluación

Define qué significa “funcionar bien”. Una evaluación puede medir exactitud, fidelidad a fuentes, selección de herramientas, coste, latencia, seguridad o utilidad para el usuario. Sin evaluación, el equipo solo posee impresiones.

8.9 Supervisión humana

La persona no debe aparecer únicamente al final para asumir la responsabilidad. El sistema debe incorporar puntos claros donde el usuario revisa, corrige o autoriza. La supervisión es parte de la arquitectura, no un parche posterior.

9. La regla de mínima complejidad suficiente

Una de las decisiones más importantes de AI Engineering es evitar que la fascinación tecnológica sustituya al análisis del problema. Anthropic señala que muchos casos se resuelven adecuadamente optimizando una sola llamada al modelo, añadiendo recuperación de información o utilizando ejemplos en contexto; los sistemas agénticos suelen intercambiar mayor coste y latencia por flexibilidad y desempeño.[1]

  1. Primero, lógica tradicional: reglas, filtros, consultas SQL y validaciones deterministas.
  2. Luego, una llamada al modelo: para la parte lingüística o ambigua.
  3. Después, RAG y herramientas: si necesita conocimiento o acciones externas.
  4. Más tarde, workflow: si la tarea tiene varias etapas conocidas.
  5. Agente: cuando el recorrido no puede predefinirse.
  6. Multiagente: cuando la separación aporta una ventaja verificable.
Pregunta de arquitectura ¿Qué capacidad concreta obtenemos al añadir este nivel de complejidad y cómo demostraremos que mejora el resultado?

10. Caso aplicado: asistente documental para estudios por imágenes

Utilizaremos un ejemplo cercano a procesos reales, pero diseñado con datos ficticios y sin información personal de pacientes.

10.1 Problema

El personal administrativo recibe solicitudes expresadas de formas diferentes. Debe identificar el estudio, región, modalidad, contraste, financiador, código y valor vigente. Los errores pueden surgir por sinónimos, documentos desactualizados o reglas diferentes entre financiadores.

10.2 Solución mínima

La primera versión no necesita un sistema multiagente. Puede utilizar:

  • Interfaz para escribir o seleccionar el estudio.
  • Backend que normaliza la consulta.
  • RAG sobre una base maestra versionada.
  • Modelo que interpreta variantes lingüísticas.
  • Reglas deterministas para cálculos.
  • Respuesta con fuente, fecha de vigencia y advertencias.
  • Registro de consultas para detectar nuevos sinónimos.

10.3 Qué debe hacer el modelo

Interpretar la forma en que el usuario expresa el estudio, relacionar sinónimos, pedir aclaraciones cuando falten datos y redactar una respuesta comprensible.

10.4 Qué no debe dejarse al modelo

Los cálculos aritméticos, la vigencia de una tabla, la autorización para modificar valores y la decisión de utilizar información sensible deben quedar controlados por código y permisos.

10.5 Cuándo aparecería un agente

Un agente podría justificarse más adelante para investigar discrepancias entre documentos, proponer actualizaciones, preparar un informe de inconsistencias y solicitar aprobación. No debería modificar la fuente maestra por sí mismo.

10.6 Métricas

  • Porcentaje de consultas correctamente identificadas.
  • Porcentaje de respuestas con fuente vigente.
  • Número de aclaraciones necesarias.
  • Tiempo promedio de resolución.
  • Errores de cálculo.
  • Consultas sin evidencia suficiente.

12. Cápsula de entrevista técnica

Pregunta: ¿Cuál es la diferencia entre un workflow y un agente? Un workflow sigue una ruta programada y previsible: el software determina qué paso viene después. En un agente, el modelo decide dinámicamente el siguiente paso y qué herramienta utilizar según los resultados que obtiene. Prefiero workflows para procesos repetibles y agentes para problemas abiertos donde no puedo anticipar la secuencia completa. En ambos casos incorporo límites, evaluación, trazabilidad y aprobación humana cuando existen acciones sensibles.
Pregunta: ¿Por qué no utilizar siempre varios agentes? Porque cada agente adicional aumenta llamadas, coste, latencia, transferencia de contexto y dificultad de depuración. Solo separo agentes cuando existe una ventaja concreta: especialización, aislamiento de permisos, ejecución paralela o evaluación independiente.
Pregunta: ¿Qué convierte a un modelo en un producto? La combinación con interfaz, lógica de negocio, datos, herramientas, persistencia, seguridad, observabilidad y una evaluación vinculada al objetivo del usuario. El modelo es una capacidad; el producto es el sistema completo.

13. Actividad práctica

Mapa de un proceso real

Selecciona un proceso que conozcas bien. Puede ser administrativo, educativo o de desarrollo de software.

  1. Describe el resultado final esperado.
  2. Enumera las etapas que hoy realiza una persona.
  3. Marca cuáles son deterministas y cuáles contienen ambigüedad.
  4. Decide dónde usarías código tradicional.
  5. Decide dónde usarías un modelo.
  6. Identifica qué datos o herramientas externas necesita.
  7. Define qué acciones requieren aprobación.
  8. Propón tres métricas para evaluar el sistema.
  9. Determina si basta una llamada, un workflow o un agente.
