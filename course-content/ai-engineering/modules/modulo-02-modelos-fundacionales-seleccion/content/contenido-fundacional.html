1. Propósito del módulo

En el Módulo 1 aprendimos que un modelo es una capacidad dentro de un sistema. Ahora debemos responder una pregunta decisiva para cualquier producto de IA: ¿qué modelo conviene utilizar para cada función?

La elección no se resuelve preguntando cuál es “el modelo más inteligente”. En producción, una solución puede fracasar aun usando un modelo excelente si responde demasiado lento, resulta demasiado caro, no admite la modalidad necesaria, tiene condiciones de uso incompatibles, cambia de versión sin control o no alcanza el nivel de privacidad requerido.

Del mismo modo, una solución puede ser profesional utilizando modelos pequeños o especializados si estos cumplen con precisión una tarea limitada. En AI Engineering, seleccionar un modelo significa equilibrar capacidades técnicas, restricciones operativas, riesgos y valor de negocio.

Idea central No existe un mejor modelo en términos absolutos. Existe un modelo más adecuado para una tarea, unos datos, un nivel de riesgo, una experiencia de usuario y un presupuesto concretos.

Objetivos de aprendizaje

  • Explicar qué convierte a un modelo en modelo fundacional.
  • Distinguir modelos generales, especializados, propietarios, abiertos y open-weight.
  • Evaluar modalidades, contexto, latencia, coste, disponibilidad, privacidad y gobernanza.
  • Interpretar benchmarks sin convertirlos en una decisión automática.
  • Diseñar evaluaciones propias basadas en tareas reales.
  • Construir una matriz profesional de selección.
  • Asignar modelos distintos a capacidades diferentes del JSG AI Engineering Hub.

2. Qué es un modelo fundacional

El término foundation model fue popularizado para describir modelos entrenados con datos amplios y adaptables a numerosas tareas posteriores. La palabra “fundacional” no indica que el modelo sea un producto terminado; señala que puede servir de base para construir muchas aplicaciones y especializaciones.[1]

Modelo fundacional
Modelo entrenado sobre datos amplios, normalmente a gran escala, que puede adaptarse mediante instrucciones, ejemplos, recuperación de información, herramientas o ajuste adicional para resolver múltiples tareas.

2.1 Una capacidad amplia, pero incompleta

Un modelo fundacional puede redactar, clasificar, resumir, interpretar imágenes, generar código o relacionar información. Sin embargo, sigue siendo un activo intermedio. Para convertirse en una solución necesita datos del dominio, controles, evaluación y una experiencia de uso.

La amplitud trae una ventaja y un riesgo. La ventaja es la reutilización: una sola familia puede sostener numerosas aplicaciones. El riesgo es la homogeneización: si muchas soluciones dependen del mismo modelo, sus limitaciones, sesgos o fallas pueden propagarse a todos los productos construidos sobre él.[1]

2.2 Preentrenamiento, adaptación e inferencia

Preentrenamiento amplio→Adaptación al uso→Inferencia en producción
  • Preentrenamiento: el modelo aprende patrones generales a partir de grandes conjuntos de datos.
  • Adaptación: el sistema lo orienta mediante prompts, ejemplos, fine-tuning, RAG, herramientas o políticas.
  • Inferencia: el modelo procesa solicitudes reales dentro de una aplicación.
Decisión de arquitectura En la mayoría de los proyectos, el equipo no necesita entrenar un modelo fundacional. Necesita seleccionar uno, adaptarlo de forma controlada y evaluar su comportamiento dentro del sistema completo.

3. Familias de modelos y modalidades

El mercado actual no contiene únicamente “modelos de texto”. Existen familias y servicios orientados a modalidades y tareas diferentes. La primera decisión es determinar qué tipo de información entra al sistema y qué salida debe producir.

Familia o modalidadEntradaSalidaAplicaciones típicas
LenguajeTextoTexto o datos estructuradosRedacción, clasificación, extracción, soporte, código
MultimodalTexto, imágenes, documentos, audio o videoTexto, análisis o contenido multimodalComprensión documental, inspección visual, asistentes ricos
EmbeddingsTexto, imagen u otras modalidadesVectoresBúsqueda semántica, RAG, agrupamiento y similitud
Voz y audioAudio o textoTranscripción, voz o audioAsistentes de voz, accesibilidad, traducción y narración
Imagen y video generativoTexto, imágenes o videoImagen o videoDiseño, educación, publicidad y prototipado visual
Moderación o seguridadContenidoClasificación de riesgoControles de contenido y políticas
Modelos especializadosDominio o tarea concretaResultado especializadoCódigo, investigación, robótica, clasificación o extracción

Los catálogos oficiales de los proveedores muestran esta especialización: modelos rápidos y económicos, modelos de razonamiento, modelos de audio, embeddings, visión y agentes gestionados pueden coexistir dentro de un mismo ecosistema.[2] [3] [4]

3.1 Modalidad requerida frente a modalidad disponible

Que un modelo acepte imágenes no significa que sea ideal para análisis clínico de imágenes, ni que un modelo acepte audio significa que sea adecuado para transcripción en ambientes ruidosos. La modalidad es una condición necesaria, no una garantía de calidad.

Ejemplo Un sistema que recibe pedidos médicos escaneados puede requerir OCR o visión para extraer texto, un modelo de lenguaje para interpretar variantes, embeddings para recuperar códigos y reglas deterministas para calcular valores. Una sola llamada multimodal no necesariamente sustituye esa arquitectura.

4. Modelos propietarios, abiertos y open-weight

La selección también depende del nivel de acceso y control que ofrece el proveedor o la comunidad.

4.1 Modelos propietarios mediante API

El proveedor opera el modelo y expone una interfaz de uso. El equipo consumidor no administra pesos ni infraestructura de inferencia.

  • Ventajas: acceso rápido, modelos de frontera, operación administrada, escalabilidad y herramientas integradas.
  • Limitaciones: dependencia del proveedor, costes variables, cambios de versión, restricciones contractuales y menor control técnico.

4.2 Modelos open-weight

Los pesos pueden descargarse o ejecutarse en infraestructura propia, aunque la licencia y la información disponible no siempre conceden todas las libertades de un sistema plenamente abierto. Por eso “pesos disponibles” y “open source” no son sinónimos.

  • Ventajas: despliegue privado, personalización, control de versiones y posibilidad de optimización.
  • Limitaciones: infraestructura, seguridad, mantenimiento, evaluación y operación pasan a ser responsabilidad del equipo.

4.3 Open Source AI

La definición de Open Source AI de la Open Source Initiative exige libertades para usar, estudiar, modificar y compartir, además del acceso a la forma preferida para modificar el sistema, que incluye información suficiente sobre datos, código y parámetros.[5]

Tipo de accesoPesosOperación propiaControlResponsabilidad operativa
API propietariaNoNoLimitado por contrato y APICompartida con el proveedor

Error frecuente Elegir un modelo local “por privacidad” sin calcular quién actualizará dependencias, controlará vulnerabilidades, medirá calidad, administrará GPU, registrará incidentes y garantizará disponibilidad.

5. Modelos generales frente a modelos especializados

Un modelo generalista cubre muchas tareas y facilita el prototipado. Un modelo especializado puede ofrecer mejor relación entre calidad, coste y control cuando la función está bien delimitada.

5.1 Cuándo conviene un modelo general

  • La tarea cambia con frecuencia.
  • Se necesita razonamiento entre dominios.
  • El producto aún está en descubrimiento.
  • Se requiere una experiencia conversacional amplia.
  • La misma llamada debe combinar texto, visión, código o herramientas.

5.2 Cuándo conviene un modelo especializado

  • La tarea es estrecha y repetitiva.
  • Existe un criterio de salida claro.
  • El volumen es alto y el coste importa.
  • La latencia debe ser mínima.
  • La evaluación demuestra que un modelo menor alcanza el umbral requerido.
Principio profesional Usar un modelo más pequeño no es “bajar la calidad” si la tarea está bien definida y el resultado supera la evaluación exigida.

5.3 Especialización por función, no por moda

En un producto puede haber un modelo de clasificación, otro para planificación, un modelo de embeddings y un modelo multimodal. La especialización debe surgir de la arquitectura y de mediciones, no de la cantidad de marcas que queremos exhibir.

6. Tamaño, capacidad y razonamiento

Durante años, el número de parámetros se utilizó como señal indirecta de capacidad. Hoy resulta insuficiente. Distintos métodos de entrenamiento, arquitecturas, mezcla de expertos, datos, herramientas y técnicas de inferencia hacen que dos modelos de tamaño similar se comporten de manera muy diferente.

6.1 “Más grande” no es una especificación de producto

Un equipo debe preguntar:

  • ¿Resuelve nuestras tareas?
  • ¿Sigue instrucciones y formatos?
  • ¿Utiliza herramientas con estabilidad?
  • ¿Mantiene calidad en español y en nuestro vocabulario?
  • ¿Produce respuestas verificables?
  • ¿Cuál es su tasa de error en casos difíciles?

6.2 Modelos de razonamiento

Algunos modelos emplean mayor cómputo de inferencia para tareas complejas. Pueden ser útiles en planificación, matemáticas, código o análisis con múltiples restricciones, pero suelen aumentar latencia y coste. No es eficiente asignarlos automáticamente a saludos, extracción simple o clasificación rutinaria.

Regla de asignación Reserva la mayor capacidad para los pasos donde el error o la ambigüedad lo justifican. Utiliza modelos rápidos o código determinista en pasos simples.

6.3 Capacidad efectiva dentro del sistema

La calidad observable no depende solo del modelo. Depende de instrucciones, contexto, recuperación, herramientas, formato de salida y evaluación. Un modelo moderado con contexto correcto puede superar a uno más capaz que trabaja sin evidencia o con un prompt deficiente.

7. Ventana de contexto y límites reales

La ventana de contexto indica la cantidad máxima aproximada de información que puede procesarse en una interacción. Incluye instrucciones, mensajes, documentos, resultados de herramientas y la salida generada.

Ventana de contexto
Presupuesto de tokens disponible para la información de entrada, el estado de la conversación y, según el proveedor, parte o toda la salida.

7.1 Capacidad máxima no equivale a calidad constante

Un modelo puede aceptar un contexto muy largo y aun así perder precisión al localizar detalles, relacionar fragmentos distantes o distinguir información relevante de ruido. La ventana anunciada es un límite técnico; la calidad útil debe medirse con documentos y preguntas reales.

7.2 Coste de enviar demasiado contexto

  • Más tokens y mayor coste.
  • Mayor latencia.
  • Más información irrelevante.
  • Riesgo de instrucciones contradictorias.
  • Dificultad para observar qué evidencia influyó.

7.3 Context engineering

La ingeniería de contexto consiste en seleccionar y organizar la información necesaria para cada paso. En lugar de enviar todo el repositorio, toda la conversación o todos los documentos, el sistema recupera solo lo relevante, resume estado y conserva trazabilidad.

Aplicación al curso Codex pudo continuar el proyecto entre hilos porque el repositorio, Git, AGENTS.md y los manifiestos conservaron el estado. No fue necesario mantener toda la conversación original dentro de cada contexto.

8. Latencia, coste, disponibilidad y ciclo de vida

Los proveedores suelen presentar familias diferenciadas por capacidad, velocidad y coste. Esa segmentación confirma que la selección es una decisión multidimensional, no una clasificación única de “mejor a peor”.[2] [3] [4]

8.1 Latencia

Debemos distinguir:

  • Tiempo hasta el primer token: cuánto tarda el usuario en percibir respuesta.
  • Velocidad de generación: cuánto tarda la salida completa.
  • Latencia del workflow: suma de modelo, RAG, herramientas, red y validaciones.

Un modelo excelente puede producir una mala experiencia si el usuario espera demasiado. En procesos no interactivos, en cambio, la calidad puede importar más que la respuesta inmediata.

8.2 Coste total

El precio por token es solo una parte. El coste total incluye:

  • entradas y salidas;
  • reintentos;
  • contextos demasiado grandes;
  • embeddings y almacenamiento;
  • herramientas externas;
  • observabilidad y evaluación;
  • infraestructura local;
  • tiempo humano de supervisión;
  • coste de los errores.

8.3 Disponibilidad, límites y regiones

La decisión debe revisar disponibilidad geográfica, cuotas, rate limits, concurrencia, acuerdos de servicio, soporte y posibilidad de operar en las regiones requeridas.

8.4 Versiones estables, preview y deprecaciones

Los proveedores pueden ofrecer versiones estables, preview, experimentales o alias que cambian con el tiempo. Para producción conviene fijar versiones cuando sea posible, registrar evaluaciones por versión y preparar migraciones. La documentación de Gemini, por ejemplo, distingue versiones estables, preview, latest y experimentales y recomienda versiones estables específicas para la mayoría de aplicaciones de producción.[4]

Riesgo de producción Usar un alias “latest” sin evaluaciones automáticas puede cambiar el comportamiento del producto aunque el equipo no haya modificado su código.

9. Benchmarks, leaderboards y evaluación propia

Los benchmarks permiten comparar modelos en condiciones estandarizadas. Son útiles para descubrir candidatos, pero no sustituyen la evaluación de la aplicación.

9.1 Qué puede decir un benchmark

  • Rendimiento relativo en una tarea definida.
  • Capacidades generales de razonamiento, lenguaje, código o visión.
  • Evolución entre versiones.
  • Fortalezas y debilidades preliminares.

9.2 Qué no garantiza

  • Calidad en tus documentos y usuarios.
  • Cumplimiento de formato.
  • Estabilidad en tool calling.
  • Precisión con terminología local.
  • Coste total del workflow.
  • Seguridad en un contexto sensible.

El informe de Stanford sobre modelos fundacionales sostiene que la evaluación debe ir más allá de exactitud e incluir robustez, equidad, eficiencia y otros factores. HELM fue creado precisamente para una evaluación más amplia y transparente.[1] [6]

9.3 Evaluación basada en tareas reales

Un conjunto de evaluación profesional debe incluir:

  1. Casos representativos del uso diario.
  2. Casos límite y ambiguos.
  3. Resultados esperados o criterios de aceptación.
  4. Errores críticos diferenciados de errores menores.
  5. Medición de coste y latencia.
  6. Revisión humana donde no exista una respuesta única.
  7. Repetición por versión de modelo y prompt.
CalidadExactitud, utilidad, formato y consistencia.
OperaciónLatencia, disponibilidad y throughput.
EconomíaCoste por tarea exitosa, no solo por token.
RiesgoPrivacidad, seguridad, sesgos y control.

9.4 El coste por tarea exitosa

Un modelo barato puede resultar caro si obliga a repetir solicitudes o genera errores. Una métrica útil es:

Coste total de intentos÷Tareas aceptadas=Coste por tarea exitosa

10. Privacidad, residencia y control de datos

La selección debe considerar qué información recibirá el modelo y qué obligaciones tiene la organización. La pregunta correcta no es solo “¿el proveedor usa mis datos para entrenar?”, sino un conjunto más amplio de controles.

10.1 Preguntas necesarias

  • ¿Dónde se procesan y almacenan los datos?
  • ¿Cuánto tiempo se retienen?
  • ¿Se utilizan para mejorar modelos?
  • ¿Qué configuraciones contractuales existen?
  • ¿Hay cifrado y aislamiento adecuados?
  • ¿Se pueden auditar accesos?
  • ¿Qué subprocesadores intervienen?
  • ¿Existe opción regional, privada o local?

10.2 Datos sensibles y minimización

Antes de cambiar de proveedor o ejecutar un modelo local, debemos reducir la información enviada. La minimización, anonimización, control de permisos y separación de funciones son decisiones de arquitectura.

10.3 Gobierno del riesgo

El AI Risk Management Framework de NIST propone tratar el riesgo de IA mediante funciones de gobernanza, identificación, medición y gestión. La selección del modelo debe integrarse a ese proceso y no quedar como una decisión aislada del desarrollador.[7]

Aplicación clínica Un sistema administrativo puede usar un modelo externo para interpretar un pedido anonimizado y reglas locales para calcular valores. No necesita enviar nombre, documento, historia clínica ni otros datos personales si esos elementos no son necesarios para la tarea.

11. Model routing y estrategias híbridas

Un sistema profesional no necesita utilizar el mismo modelo para todas las solicitudes. El routing asigna cada tarea al modelo o mecanismo más adecuado según criterios definidos.

11.1 Tipos de routing

  • Por tarea: clasificación, extracción, planificación o generación.
  • Por complejidad: modelo rápido para casos simples y modelo avanzado para ambiguos.
  • Por modalidad: texto, visión, audio o embeddings.
  • Por sensibilidad: local o privado para datos restringidos.
  • Por disponibilidad: fallback cuando un proveedor falla.
  • Por coste: presupuesto máximo por tarea.

11.2 Arquitectura en cascada

Regla determinista→Modelo rápido→Modelo avanzado→Revisión humana

La cascada evita pagar el nivel más caro para todos los casos. El escalamiento debe basarse en señales observables: baja confianza, formato inválido, conflicto documental o riesgo elevado.

11.3 Fallback no significa equivalencia

Un proveedor alternativo puede usar prompts, formatos y herramientas diferentes. La conmutación requiere una capa de abstracción, evaluaciones y adaptación. No basta con cambiar el nombre del modelo.

Principio La estrategia multimodelo aporta resiliencia y eficiencia solo cuando el coste de coordinación es menor que el beneficio obtenido.

12. Matriz profesional de selección

La matriz convierte preferencias generales en una decisión documentada. Antes de comparar proveedores, define el umbral mínimo de cada criterio.

CriterioPreguntaMétrica o evidenciaPeso sugerido
Calidad de tarea¿Resuelve casos reales?Evaluación propia, tasa de aceptación20–30 %
Fiabilidad¿Respeta formato y herramientas?Errores de esquema, tool calls correctas10–20 %
Latencia¿Cumple la experiencia?P50, P95 y tiempo total5–15 %
Coste¿Es sostenible?Coste por tarea exitosa5–15 %
Contexto¿Procesa la evidencia necesaria?Pruebas de recuperación y lectura larga5–10 %
Modalidad¿Acepta entradas requeridas?Texto, visión, audio, documentosObligatorio
Privacidad¿Cumple la política?Contrato, retención, región y controlesObligatorio o 10–25 %
Disponibilidad¿Puede operar al volumen esperado?Cuotas, SLA, regiones y fallback5–10 %
Control de versión¿Podemos reproducir resultados?Versionado, deprecación y migración5–10 %
Licencia y portabilidad¿Podemos desplegar o modificar?Licencia, pesos, código y dependenciaSegún estrategia

12.1 Proceso de decisión

  1. Define la tarea y el resultado aceptable.
  2. Identifica restricciones obligatorias.
  3. Selecciona tres a cinco candidatos.
  4. Construye un conjunto de evaluación.
  5. Mide calidad, coste, latencia y errores.
  6. Documenta la decisión y sus supuestos.
  7. Define cuándo reevaluar.
Documento de decisión La matriz no debe terminar en “ganó el modelo X”. Debe registrar por qué ganó, en qué versión, con qué prompts, sobre qué casos y bajo qué límites.

13. Caso aplicado: selección para un asistente documental

Consideremos un asistente que recibe una consulta sobre un estudio por imágenes y debe identificar modalidad, región, contraste, financiador, código y valor vigente.

13.1 Descomposición de capacidades

CapacidadOpción adecuadaJustificación
Detectar modalidad y regiónModelo rápido o clasificadorTarea limitada, alto volumen y salida estructurada
Interpretar solicitud ambiguaModelo general más capazRequiere lenguaje, contexto y aclaraciones
Recuperar códigos y reglasEmbeddings + RAGLa evidencia debe venir de la base maestra
Calcular valoresCódigo deterministaLa aritmética y reglas deben ser reproducibles
Procesar documento escaneadoOCR o modelo multimodalLa entrada no está disponible como texto limpio
Información sensibleMinimización o modelo privadoReducir exposición y cumplir políticas

13.2 Selección inicial

El equipo puede comenzar con un modelo general de velocidad media para interpretar consultas y un modelo de embeddings para recuperación. Solo incorporará un modelo de razonamiento avanzado si las evaluaciones muestran que los casos ambiguos no alcanzan el umbral.

13.3 Qué medir

  • Exactitud al identificar estudio y región.
  • Capacidad para solicitar aclaraciones.
  • Uso correcto de las fuentes recuperadas.
  • Errores críticos de código o valor.
  • Latencia total.
  • Coste por consulta aceptada.
  • Porcentaje derivado a revisión humana.
Resultado arquitectónico La selección no produce “un modelo ganador”. Produce una combinación: modelo rápido, modelo de escalamiento, embeddings, código determinista y supervisión humana.

15. Cápsula de entrevista técnica

Pregunta: ¿Cómo seleccionas un modelo para producción? Primero defino la tarea, los datos, el riesgo y el umbral de aceptación. Después comparo candidatos mediante evaluaciones propias y mido calidad, latencia, coste por tarea exitosa, privacidad, disponibilidad y estabilidad de versión. No decido únicamente por benchmarks.
Pregunta: ¿Por qué no usar siempre el modelo más capaz? Porque muchas tareas no necesitan esa capacidad. Un modelo mayor puede aumentar coste y latencia sin mejorar el resultado. Conviene asignar modelos según la complejidad y escalar solo cuando una evaluación demuestra que es necesario.
Pregunta: ¿Qué diferencia existe entre open-weight y open source? Open-weight significa que los parámetros pueden estar disponibles, pero no garantiza acceso suficiente a datos, código, proceso de entrenamiento y libertades de modificación. La definición Open Source AI exige condiciones más amplias.
Pregunta: ¿Qué es model routing? Es la capa que decide qué modelo, herramienta o ruta debe atender cada solicitud según tarea, complejidad, modalidad, sensibilidad, coste o disponibilidad.

Frases clave

  • “El benchmark propone candidatos; la evaluación propia toma la decisión”.
  • “La ventana máxima de contexto no es la cantidad de contexto que deberíamos enviar”.
  • “El coste relevante es el coste por tarea exitosa”.
  • “Un sistema profesional puede usar varios modelos sin convertirse en un sistema multiagente”.

16. Actividad práctica — Matriz de selección

Selecciona un proceso real que conozcas. Puede ser administrativo, educativo, clínico o de desarrollo de software.

Parte A. Definición

  1. Describe el resultado final esperado.
  2. Enumera las entradas y modalidades.
  3. Indica qué datos son sensibles.
  4. Define el tiempo máximo aceptable.
  5. Identifica errores críticos.

Parte B. Descomposición

Separa el proceso en capacidades:

  • clasificación;
  • extracción;
  • recuperación;
  • razonamiento;
  • generación;
  • cálculo;
  • validación;
  • acción externa.

Parte C. Candidatos

Propón al menos dos arquitecturas:

  • Arquitectura simple: un modelo general y código.
  • Arquitectura optimizada: routing entre modelos o funciones especializadas.

Parte D. Matriz
