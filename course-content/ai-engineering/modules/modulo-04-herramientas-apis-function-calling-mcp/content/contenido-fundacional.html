1. Propósito del módulo

Un modelo puede razonar y generar texto, pero no puede consultar un calendario, crear un registro, leer una base privada o enviar una notificación sin una integración que conecte el razonamiento con el mundo exterior.

Este módulo estudia cómo convertir capacidades externas en herramientas seguras y comprensibles para un sistema de IA. Diferenciaremos una API tradicional, una herramienta, el ciclo de function calling y el Model Context Protocol.

Idea centralEl modelo no ejecuta directamente una API. Propone una llamada estructurada; la aplicación valida, autoriza, ejecuta y devuelve el resultado.

Objetivos

  • Distinguir modelo, herramienta, función, API, conector y servidor MCP.
  • Comprender el ciclo completo de function calling.
  • Diseñar contratos de herramientas mediante JSON Schema.
  • Decidir cuándo usar integración directa y cuándo MCP.
  • Aplicar validación, permisos, idempotencia, timeouts y aprobación humana.
  • Diseñar la capa de herramientas del JSG AI Engineering Hub v0.4.

2. Del modelo al mundo exterior

Una aplicación inteligente combina razonamiento probabilístico con ejecución determinista.

Usuario→Modelo decide→Llamada estructurada→Aplicación valida→API o servicio→Resultado→Modelo responde
Error frecuenteConfundir “el modelo pidió una acción” con “la acción ya fue ejecutada”. Entre ambos eventos debe existir una capa de control.

ConceptoFunciónEjemplo
APIContrato técnico para comunicarse con un servicio.Consultar eventos de un calendario.
FunciónUnidad de código ejecutable dentro de la aplicación.buscar_turnos(fecha).
HerramientaCapacidad descrita al modelo con nombre, propósito y esquema.Buscar disponibilidad.
ConectorIntegración empaquetada con autenticación y acceso.Drive, correo o CRM.

PrincipioUna API está diseñada para programas. Una herramienta debe estar diseñada también para que el modelo comprenda cuándo usarla y con qué argumentos.

4. El ciclo de function calling

Function calling es un patrón de orquestación: el modelo produce una solicitud estructurada; el runtime ejecuta código y devuelve el resultado para que el modelo continúe.

  1. La aplicación declara herramientas.
  2. El usuario solicita una tarea.
  3. El modelo selecciona una herramienta y genera argumentos.
  4. La aplicación valida esquema y permisos.
  5. El runtime ejecuta la función o API.
  6. El resultado vuelve al modelo.
  7. El modelo responde o solicita otra herramienta.
tools+prompt→tool call→execute→tool result→final answer

6. Selección y carga de herramientas

Dar al modelo demasiadas herramientas aumenta tokens y ambigüedad. Conviene conservar un conjunto relevante, agrupar por namespaces y cargar herramientas bajo demanda cuando el catálogo es grande.

  • Agrupar por dominio.
  • Evitar funciones superpuestas.
  • Describir límites y efectos.
  • Usar carga diferida.
  • Medir selección correcta.

9. Integración directa con APIs

Es apropiada cuando la aplicación controla proveedor, contrato y runtime. Ofrece menor complejidad para pocas herramientas, pero aumenta mantenimiento y acoplamiento cuando crece el número de proveedores.

10. Qué es MCP

Model Context Protocol es un protocolo abierto que estandariza cómo una aplicación de IA se conecta con fuentes de datos y capacidades externas.

HostAplicación de IA que administra conexiones, permisos y experiencia.
Cliente MCPComponente del host que mantiene una conexión con un servidor.
Servidor MCPServicio que expone capacidades y contexto.

MCP utiliza JSON-RPC 2.0, negociación de versión y capacidades. No decide cómo razona el modelo; estandariza el intercambio.

12. Transporte, ciclo de vida y autorización

  • stdio: comunicación entre procesos locales.
  • Streamable HTTP: comunicación remota mediante HTTP y streaming opcional.
Inicialización→Negociar versión→Declarar capacidades→Operar→Cerrar

Para servidores HTTP protegidos, MCP define autorización basada en OAuth 2.1. Los servidores locales por stdio suelen obtener credenciales del entorno.

14. Seguridad

  • Validar entradas.
  • Aplicar mínimo privilegio.
  • No confiar ciegamente en servidores externos.
  • Mostrar acciones sensibles.
  • Sanear resultados.
  • Aplicar límites, timeouts y auditoría.
  • Evitar passthrough inseguro de tokens.
  • Separar secretos del contexto.
  • Protegerse contra prompt injection y exfiltración.
Superficie de ataqueUna herramienta convierte texto generado por un modelo en una posible acción real. La seguridad debe estar en el runtime y en el servicio, no solo en el prompt.

16. Caso aplicado

“Busca disponibilidad para una resonancia y prepara un correo, pero no lo envíes”.

  1. El sistema identifica consulta y borrador.
  2. Usa buscar_disponibilidad.
  3. Valida fecha y permisos.
  4. Recibe horarios.
  5. Usa crear_borrador_correo.
  6. No usa enviar_correo.
  7. Registra la traza.
AprendizajeUna tarea útil requiere selección, contrato, permisos y control de efectos; no solo acceso a una API.

18. Cápsula de entrevista

¿El modelo ejecuta una función?No. Produce una llamada estructurada y el runtime decide si la ejecuta.
¿Qué aporta JSON Schema?Un contrato validable de argumentos.
¿Cuándo usar MCP?Cuando se busca interoperabilidad y reutilización entre hosts.
¿Tool o resource?La tool ejecuta; el resource aporta datos.
¿Dónde se aplica la seguridad?En runtime, políticas y servicio externo.
